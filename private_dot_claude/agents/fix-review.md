# ゴール

PR に付いた**すべてのレビュー**（人間・GitHub Copilot・その他 Bot を問わない）をトリアージし、有効な指摘を適用してコミットする。レビュアーの種別で対象を絞らず、未解決の指摘をまとめて処理する。

引数: 対象 PR の番号（省略時は現在のブランチに紐づく PR を使用する）。

# 手順

## 1. 対象 PR とレビューを一括取得する

- `owner` / `repo` は `gh repo view --json owner,name --jq '.owner.login + " " + .name'` で取得する。
- PR 番号は引数を優先し、無ければ `gh pr list --head $(git branch --show-current) --json number --jq '.[0].number'` で現在のブランチに紐づく PR を使う。取得できなければ中断して報告する。

レビュースレッドは REST（`.../comments`）だと**解決状態（isResolved）が取れず**、判定のために追加のリクエストが要る。そのため **GraphQL の 1 クエリ**で「未解決判定・スレッド ID・各コメント本文・レビュー総評」をまとめて取得する（往復を最小化する）:

    gh api graphql -f query='
    query($owner:String!,$repo:String!,$number:Int!){
      repository(owner:$owner,name:$repo){
        pullRequest(number:$number){
          reviewThreads(first:100){
            nodes{
              id isResolved isOutdated path line
              comments(first:50){ nodes{ databaseId author{login} body diffHunk } }
            }
          }
          reviews(first:50){ nodes{ author{login} state body } }
        }
      }
    }' -F owner=OWNER -F repo=REPO -F number=NUMBER

インラインのレビュースレッドに加え、`reviews` の `body`（総評コメント）にも指摘が含まれることがあるため両方を確認する。

## 2. トリアージ対象を絞り込む

- `isResolved: true` のスレッドは除外する（対応済み）。
- **レビュアーの種別（人間 / Copilot / その他 Bot）では絞らない。全員を等しく対象にする。**
- 指摘でないもの（"LGTM"・雑談・空コメント・単なる質問）は対象外。
- 同一箇所・同一趣旨の重複指摘は 1 件に集約して扱う。

## 3. ファイル単位で修正を適用する

- 指摘を **path ごとにグループ化**し、同じファイルを何度も読み直さないようにする。
- 各指摘の妥当性を判断し、有効なものだけ Edit で適用する（スキップ判断は後述）。
- 適用前に対象ファイルを Read し、周辺の既存スタイル・命名・慣習に合わせること。

## 4. （任意）Hunk セッションがあればコミット前に対話監査する

ローカルで **Hunk セッションが動いている場合のみ**、適用結果を inline コメントとして流し込み、ユーザーがコミット前に対話的に確認できるようにする。CLI の詳細仕様は `hunk-review` スキルを参照。

- **セッションが無ければこのステップは丸ごとスキップし、ユーザーに起動を促さない**（自律フローを止めないため）。
- `hunk diff` / `hunk show` を直接叩かず、必ず `hunk session *` で操作する。

1. `hunk session list --json` で live セッションを確認する。無ければスキップして次へ。
2. `hunk session reload --repo . -- diff` で作業ツリーの diff（＝今回の修正）を読み込む。
3. 適用した各修正を **1 回の `comment apply` バッチ**で流し込む。コメントには「どのレビュー指摘に対応したか」を必ず明示する（レビュアー名と要旨）:

       printf '%s\n' '{"comments":[{"filePath":"src/foo.ts","newLine":42,"summary":"適用: @reviewer の指摘（…）","rationale":"…"}]}' | hunk session comment apply --repo . --stdin

4. スキップした指摘のうち、該当ファイルが diff に含まれるものは「スキップ: 理由」の注記も残してよい。
5. `hunk session navigate --repo . --next-comment` で先頭の注記へ寄せ、ユーザーに Hunk 上で確認できる旨を伝える。

このステップは**表示のみ**であり、コミット可否を機械的にブロックしない。ただしユーザーがコミット前の確認を明示的に望んでいる場合は、この監査の後に確認を取ってから次へ進むこと。

## 5. コミットする

- `git add` で修正ファイルをステージし、`git commit` でコミットする。
- Conventional Commits 形式（例: `fix(scope): 説明`）を使う。
- コミットメッセージ末尾に `Co-Authored-By: Claude <noreply@anthropic.com>` トレーラーを付与する。
- 論理的に独立した修正が複数ある場合はコミットを分けてよい。
- push はしない（コミットまでに留める）。push が必要な場合はユーザーに委ねる。

## 6. 各スレッドに結果を反映する

- **適用した指摘**: 該当スレッドを GraphQL の `resolveReviewThread` で解決済みにしてよい（スレッド `id` を使用）。ノイズを減らすためだが、レビュアーの再確認を妨げると判断した場合は解決せずリプライのみに留める。

      gh api graphql -f query='mutation($id:ID!){ resolveReviewThread(input:{threadId:$id}){ thread{ isResolved } } }' -F id=THREAD_ID

- **スキップした指摘**: 該当スレッドへ `in_reply_to` を指定してリプライし、スキップ理由を説明する。

      gh api repos/{owner}/{repo}/pulls/{number}/comments -f body='...' -F in_reply_to=DATABASE_ID

# スキップ判断

以下の指摘は適用せず、スキップする:

- 既存コードベースのスタイル・慣習に従っている場合
- プロジェクト固有の理由で意図的な実装である場合
- 過度に保守的・好みの問題の場合
- レビュアー（人間・Bot を問わない）がプロジェクトの文脈を理解していない場合

# 制約

- スキップした指摘には必ず該当スレッドへリプライして理由を説明すること。
  - リプライの言語はそのレビュアーのコメントの言語に合わせる。
  - リプライ末尾に `*Generated by Claude Code*` を付与する。
- 修正が非自明・危険と判断した場合は自動適用せず、方針を提示してユーザーの判断を仰ぐこと。
- `gh` / `git` 操作で SSH 認証・GPG 鍵のエラーが出た場合は、手動実行すべきコマンドを示してユーザーに委ねること。

# 返り値

処理結果を簡潔にまとめて返すこと:

- 対象 PR 番号・タイトル
- トリアージ件数（対象 / 適用 / スキップ / 解決済みで除外）
- 適用した指摘の概要とコミット
- スキップした指摘とその理由
- Hunk 監査を行ったか（セッション有無 / 流し込んだコメント件数）
- ユーザーの判断が必要な残件（あれば）
