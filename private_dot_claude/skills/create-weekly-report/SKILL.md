# Weekly Report（Obsidian 連携）

Obsidian vault の日次 Working Log から「前週分」を集約し，週報を所定フォーマットで生成して vault に保存する．

- Vault: `~/Documents/m1sk9-vault`
- 入力: `Work/Logs/<年>/<YYYY-MM-DD>.md`（日次 Working Log）
- 出力: `Work/Weekly/<ISO年>-<ISO週番号>.md`（例 `2026-27.md`）

## 「前週」の定義

基準日（デフォルトは今日，引数で `YYYY-MM-DD` を渡せば上書き）が属する週の **1 つ前の週（月曜〜日曜）** を対象とする．
例: 2026-07-06（月）に実行 → 対象は 2026-06-29〜2026-07-05．

## 手順

### 1. 対象週と保存先ファイル名を確定する

基準日を `TODAY`（引数があればその値，無ければ `date +%Y-%m-%d`）として，以下の `date` を実行して値を得る（macOS / BSD date．シェル変数は使わず，前段の出力を次段にそのまま渡す）．

```
date -j -v-mon -v-7d -f "%Y-%m-%d" "<TODAY>" "+%Y-%m-%d"   # 前週の月曜 = PREV_MON
date -j -v-mon -v-1d -f "%Y-%m-%d" "<TODAY>" "+%Y-%m-%d"   # 前週の日曜 = PREV_SUN
date -j -f "%Y-%m-%d" "<PREV_MON>" "+%G-%V"                # ISO年-週番号 = 保存ファイル名の basename
```

保存先は `Work/Weekly/<ISO年-週番号>.md`（例 `2026-27.md`）．`%G`（ISO 年）/ `%V`（ISO 週番号, 01–53）を使うことで年跨ぎの週でも破綻しない．

### 2. 対象週の日次ログを読む

`PREV_MON` から `PREV_SUN` までの各日について，`Work/Logs/<その日付の年>/<YYYY-MM-DD>.md` を読む（読み込みは Read で行う）．

- 年サブフォルダはその**日付自身の年**を使う（年跨ぎの週は 2 つの年フォルダにまたがりうる）．
- 通常は月〜金の 5 ファイル．祝日・休暇等で欠ける日はスキップしてよい．
- **対象週のログが 1 つも見つからない場合は，その旨を報告して停止する**（空の週報は作らない）．

日次ログには 2 系統のフォーマットが混在する:

- **新テンプレ**（front matter + 見出しあり）: `## PR 処理` / `## その他作業` / `## 懸念・問題点` / `## 引き継ぎ (月〜木)` / `## 来週への引き継ぎ` などのセクションを持つ．
- **旧ログ**（front matter 無し・自由記述）: PR の URL ＋インデントした箇条書き，PR に紐づかない自由記述など．

### 3. 集約・分類する

以下のルールで内容を統合・分類する（元の日次テンプレのセクションは下記マッピングの手がかりとして使う）．

**PR 単位でまとめる．** 同じ PR が複数日に登場することは多い（レビュー対応 → 追加修正 → マージ 等）．日ごとにバラさず PR ごとに 1 エントリへ統合し，その週で結局その PR に対して何をしたかを 1〜2 文に要約する．

**PR は `repo/#<number>` 形式で表記する．** URL `https://github.com/<owner>/<repo>/pull/<number>` から `<repo>` と `<number>` を取り，`lazy-tracker/#1159` のように書く（owner と完全 URL は載せない）．番号だけ（`#1001`）で書かれていても，同じログ群から判明する repo 名を補って揃える．repo を特定できない番号はそのまま `#<number>`．

**重複・付随イベントは畳む．** 「`#1001` は `#1159` と重複のためクローズ」等は主 PR の記述に含めるか簡潔に併記し，冗長に繰り返さない．

**PR に紐づかない作業も拾う．** ツール移行・廃止・運用判断などは独立項目として書く．

**コードフェンス内の調査・分析ログは要約して使う．** 全文は転記せず，「やったこと」「懸念」「残タスク」を抽出して該当セクションへ振り分ける．

#### 分類マッピング

- **先週やったこと** — 実際に着手・完了した作業（PR 対応・マージ・実装・調査・廃止判断など過去の事実）．日次テンプレの `## PR 処理` / `## その他作業` が主な供給源．
- **今週やること** — 次週に着手する具体アクションが明確なもの．日次ログの `## 来週への引き継ぎ` / `来週:` で示された意図（「〜を棚卸したい」等）はここへ昇格させる．材料が無ければ空欄でよい．
- **懸念点** — リスク・要検証だがまだ具体アクションに落ちていない点（テスト網羅の穴，設定とコードの整合，ツール相性など）．`## 懸念・問題点` が主な供給源．今週やることへ昇格した項目をここで単純重複させない．
- **課題、今後の目標** — 残タスク・今後の方向性・要判断事項（本番投入前チェック，「〜を検討する」「独立チケット化」等の先送り・将来判断）．

迷ったら: 「放置するとまずい／見落としやすい」→ 懸念点，「いずれやる／決める」→ 課題．該当材料が無いセクションは見出しだけ残して中身は空にする（無理に埋めない）．

### 4. 週報を生成する

front matter は以下に厳密に従う．本文の 4 セクションは Obsidian ネイティブの Markdown 見出し（`## 見出し`，`#` の後ろにスペース）で書く．各項目は `- ` の箇条書き 1 行で簡潔に，元ログのインデント構造は持ち込まない．

```md
---
Categories: Work
Aliases: logs
Tags: weeklylogs
---

## 先週やったこと

- ...

## 今週やること

- ...

## 懸念点

- ...

## 課題、今後の目標

- ...
```

### 5. 保存して報告する

`Work/Weekly/<ISO年-週番号>.md` に保存する．

- **書き込みは Obsidian MCP のファイル作成／書き込みツールを使う**（グローバル規則: Obsidian への書き込みは MCP）．
- MCP が当該セッションで利用不可の場合のみ，`~/Documents/m1sk9-vault/Work/Weekly/<name>.md` へ Write でフォールバックする（その旨を報告する）．
- 同名ファイルが既に存在する場合は上書きせず，ユーザーに確認する．

保存後に以下を報告する: 対象週の期間（`PREV_MON`〜`PREV_SUN`），保存先パス，集約した日次ログの日付一覧（欠けた日があればそれも）．
