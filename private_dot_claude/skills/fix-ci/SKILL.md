# タスク: GitHub Actions 失敗の調査と修正

指定されたプルリクエストで失敗している GitHub Actions の原因を調査し，修正を適用してください．

## 手順

### 1. PR 番号の特定

引数が指定された場合は `$ARGUMENTS` を PR 番号として使用してください．
指定がない場合は以下の方法で現在の PR を特定してください：

```bash
gh pr view --json number,title,headRefName
```

### 2. PR のチェック状態を取得

以下のコマンドで PR に紐付く CI チェックの一覧と状態を取得してください：

```bash
gh pr checks {PR番号}
```

すべてのチェックが成功している場合は「すべての CI チェックが成功しています」と報告して終了してください．

### 3. 失敗しているワークフローの特定

失敗（`fail`）しているチェックを特定し，対応する workflow run を取得してください：

```bash
# PR のコミット SHA を取得
gh pr view {PR番号} --json headRefOid --jq '.headRefOid'

# そのコミットに紐付く workflow run を取得
gh run list --commit {SHA} --json databaseId,name,status,conclusion
```

### 4. 失敗ログの取得と分析

失敗した各 run について詳細なログを取得してください：

```bash
# run の詳細（失敗した job とステップの特定）
gh run view {run_id}

# 失敗した job のログを取得
gh run view {run_id} --log-failed
```

ログから以下の情報を抽出してください：

- **失敗した job 名とステップ名**
- **エラーメッセージ**（コンパイルエラー，テスト失敗，lint エラーなど）
- **該当ファイルと行番号**（ログに含まれている場合）

### 5. 失敗原因の分類

各失敗を以下のカテゴリに分類してください：

**修正可能（このスキルで対応）：**
- コンパイルエラー・型エラー
- テスト失敗（コードの変更に起因するもの）
- lint / format エラー
- 依存関係の問題（lockfile の不整合など）

**修正不可能（ユーザーに報告のみ）：**
- インフラ / ネットワークの一時的な障害
- 外部サービスの問題
- シークレット / 権限の不足
- CI 設定自体の問題（ユーザー判断が必要）
- flaky test（再現性のない失敗）

### 6. 修正の実施

修正可能な失敗について：

1. 該当ファイルを Read ツールで読み込み，現在のコードを確認する
2. エラーメッセージと該当箇所を照合して原因を特定する
3. Edit ツールで修正を適用する
4. 関連するローカルチェックがあれば実行して修正を検証する（例: `cargo check`, `npm run lint` など）

### 7. 結果の報告

以下の形式で報告してください：

```
## GitHub Actions 失敗調査結果

**PR**: #{PR番号} {タイトル}

### 失敗していた CI チェック

| ワークフロー | Job | ステップ | 原因 |
|---|---|---|---|
| {name} | {job} | {step} | {原因の要約} |

### 修正した項目 (N 件)

1. **[ファイルパス:行番号]** - {エラーの要約}
   - 原因: {なぜ失敗していたか}
   - 修正: {何をどう変えたか}

### 修正できなかった項目 (M 件)

1. **{ワークフロー / Job}** - {エラーの要約}
   - 理由: {なぜ自動修正できないか}
   - 推奨対応: {ユーザーが取るべきアクション}
```

修正を行った場合は，変更をコミットせずにユーザーに確認を促してください．
