# タスク: 不要なコメントの削除と整理

以下の手順でコメントを整理してください：

## 1. 対象ファイルの特定

引数の処理：
- 引数が指定された場合: `$ARGUMENTS`
- 引数がない場合: 最近編集されたファイル（git status、git log、または最近の会話履歴から判断）を対象とする

対象ファイルのパターン例：
- 特定のファイル: `src/app.js`
- 複数ファイル: `src/*.js`
- ディレクトリ: `src/`
- 最近開いたファイル: （引数なしの場合）

## 2. ユーザーへの確認

対象ファイルの一覧を提示し、処理を続行するか確認してください。

## 3. コメントの分類基準

### **削除すべき不要なコメント：**
- コードを読めば分かる単純な説明コメント
  - 例: `// ユーザー名を取得` の直後に `const username = user.name`
  - 例: `// ループ処理` の直後に `for (let i = 0; i < arr.length; i++)`
- 変数宣言や代入の直接的な説明
- 明らかな処理の繰り返し説明
- 古くなった・無効化されたコメント
- 意味のない区切り線のみのコメント（例: `// ----------`）

### **残すべき重要なコメント（適切なラベルを付与）：**

**TODO:** 未実装機能や今後の改善点
- 例: `// TODO: エラーハンドリングを追加する`
- 例: `// TODO: パフォーマンス最適化が必要`

**WARN:** 危険な操作や注意が必要な箇所
- 例: `// WARN: この関数は副作用があります`
- 例: `// WARN: 大量のデータを扱うため、メモリ使用量に注意`

**FIXME:** バグや修正が必要な箇所
- 例: `// FIXME: エッジケースで正しく動作しない`
- 例: `// FIXME: タイムアウト処理が不完全`

**NOTE:** 重要な補足情報や設計判断の理由
- 複雑なアルゴリズムの説明
- 特定の実装を選んだ理由
- APIの制約や制限事項
- セキュリティに関する注意事項
- 例: `// NOTE: この実装はXXX APIの制約により必要`

**HACK:** 一時的な回避策や非標準的な実装
- 例: `// HACK: ライブラリのバグを回避するための暫定対応`

## 4. 実行

各ファイルを読み込み、コメントの内容を理解した上で：
1. 不要なコメントを削除
2. 重要なコメントに適切なラベルを付与（まだない場合）
3. 既存のラベル（TODO、FIXME等）は保持
